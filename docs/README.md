# Fight & Conquer — Documentation

A Slay/Antiyoy-style turn-based hex-conquest game for Android: pure-Kotlin rules engine,
Google Filament 3D renderer inside Jetpack Compose, Blender-authored low-poly minis.

## Quickstart

```bash
# Engine tests (pure JVM, fast — the main safety net)
./gradlew :core:test

# App unit tests (mesh loader, camera projection)
./gradlew :app:testDebugUnitTest

# Build + install
./gradlew :app:assembleDebug
adb install -r app/build/outputs/apk/debug/app-debug.apk
```

### Tooling

```bash
# Drive the running Blender via MCP (add-on must be connected, see asset-pipeline.md)
python3 tools/blender_run.py exec art/blender/pieces/tree.py   # build+export a piece
python3 tools/blender_run.py thumb /tmp/preview.png            # render a thumbnail

# Bake all GLB models into game assets
python3 tools/glb2pmesh.py --all art/models app/src/main/assets/pieces

# Recompile Filament materials after editing app/src/main/materials/*.mat
MATC=/path/to/filament/bin/matc ./tools/compile-materials.sh
```

## Repository layout

```
core/                     Pure-Kotlin game engine (no Android deps) — rules, AI, maps, saves
app/                      Android app — Filament renderer + Compose HUD
  src/main/materials/     Filament material sources (.mat)
  src/main/assets/
    materials/*.filamat   Compiled materials (checked in; version-locked to Filament)
    pieces/*.pmesh        Baked piece models (generated by tools/glb2pmesh.py)
art/
  blender/_common.py      Shared bpy helpers for piece authoring
  blender/pieces/*.py     One idempotent Blender script per piece
  models/*.glb            Exported glTF models (source of the .pmesh assets)
tools/                    blender_run.py, glb2pmesh.py, compile-materials.sh
docs/                     You are here
```

## Golden rules for contributors (human or AI)

1. **`:core` never imports `android.*`.** All rules run in host JVM tests.
2. **The reducer is pure and deterministic.** RNG state lives inside `GameState`;
   never use `kotlin.random.Random`, wall clocks, or platform randomness in `:core`.
3. **Events are hints, state is truth.** The renderer animates `GameEvent`s but
   `BoardScene.reconcile()` snaps to `GameState` after every queue drain — an
   unhandled event degrades to "no animation", never a wrong board.
4. **Never assume — verify.** Rules changes need `:core` tests in the same commit;
   renderer changes need an on-device screenshot; economy/AI changes must keep the
   AI-vs-AI simulation suite green (termination, winrate, determinism).
5. Piece art changes go through the Blender pipeline (asset-pipeline.md), not by
   editing `.pmesh` bytes or bypassing the role-name convention.
